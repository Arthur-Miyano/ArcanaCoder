# 题目设计规范

> 所有新题目（ch3-ch10）创建时必须遵守。
> 违反规范的 PR 不合并。

---

## 一、基础规则

### 1.1 通用

| 规则 | 说明 | 违反示例 |
|------|------|---------|
| **TS转义感知** | TypeScript 字符串中的 `\n` 会被解释为实际换行符。需要显示 `\n` 字样时用 `\\n` | `'f"{\\n}"'` ✅ vs `'f"{\n}"'` ❌ |
| **选项互斥** | choice 题的选项必须互斥，不能有多个"逻辑上都对"的选项 | s2_02: `r""` 和 `R""` 都对但只认一个 ❌ |
| **无矛盾提示** | commonErrors 中的 hint 不能与判题逻辑矛盾 | s2_02 commonErrors: "小写r和大写R都可以" 但只认小写 ❌ |
| **知识标签完整** | 每道题至少 1 个 knowledgeTag，必须能匹配到备用题库 | |

### 1.2 选项文本（choice / output_predict）

| 规则 | 说明 |
|------|------|
| **显示一致性** | 选项文本应与 `print()` 实际输出一致。如需展示引号，前后一致 |
| **无歧义缩写** | 不出现单独看无法理解的选项（如裸 `'True'` 在非布尔题中） |

### 1.3 判题配置（validationMode）

| 模式 | 适用场景 | 行为 |
|------|---------|------|
| `exact` | 输出完全确定的题（纯 print、无用户输入） | `userOutput.trim() === expected.trim()` |
| `contains` | 输出中包含用户输入或结果不确定的场景 | `userOutput.includes(expected)` |
| `regex` | **禁止使用**。判题系统不处理 regex 模式。 | — |

**选择原则：**
- 输出完全确定 → `exact`
- 输出包含用户输入或结果不确定 → `contains`
- `input()` 在自动判题题中**禁止使用**（见 §1.4 和 §四）

### 1.4 code_fill / code_fix

| 规则 | 说明 |
|------|------|
| **禁止 input()** | 自动判题通道（code_fill/fix/free_coding）中绝对禁止使用 `input()`。包含 `input()` 的题必须标记为"人工验证" |
| **expectedOutput 必填** | 不能依赖 `runPython(correctCode)` 作为 fallback |
| **initialCode 可编辑** | 用户必须在 initialCode 基础上修改，不能是空文本框 |

### 1.6 人工验证题

需要 `input()` 的题只能走人工验证通道：

| 字段 | 要求 |
|------|------|
| 标记方式 | 在题目的 `description` 或 `narrativeDesc` 末尾标注 `[人工验证]` |
| 自动测试 | 不计入 e2e 自动测试通过率 |
| validationMode | 设为 `contains` |
| 限制 | 每章不超过 1 题人工验证题 |

### 1.5 free_coding

| 规则 | 说明 |
|------|------|
| **最少测试用例数** | ≥ 4 个 |
| **必须覆盖的边界** | 空输入、正常输入、边缘输入（极端值） |
| **测试输入为完整函数调用** | 不是裸参数 |

## 二、验证要求

### 2.1 每道新题上线前必须验证

```
□ 类型检查通过（vue-tsc）
□ 题目能在答题流中正常加载
□ choice/output_predict：4选项正确渲染
□ code_fill/fix：提交后反馈正常显示
□ 正确答案 → 魔力共鸣
□ 错误答案 → 面板显示diff/error
□ validationMode 行为符合预期
```

### 2.2 变体验证（每季度或出题批次执行）

对每道 code_fill/fix/free_coding 题验证：
- **空格变体**：首尾空格、多余空格、多余换行 → 应为 FAIL（exact模式下）
- **大小写变体**：→ 应为 FAIL（除非用 contains）
- **等价实现**：不同但正确的代码 → 应为 PASS
- **错误代码**：常见的错误模式 → 应为 FAIL + 有用错误消息

## 三、章节/节设计规范

### 3.1 章节

- 每章 18-28 题（高阶可缩减）
- 题型分布建议：
  - choice: 20-35% （识别/辨析类知识点）
  - code_fill: 30-50%（核心，培养写代码肌肉记忆）
  - output_predict: 5-15%（预测执行结果，验证理解）
  - code_fix: 5-15%（常见错误修复）
  - free_coding: 3-8%（综合应用，每章至少 1 题）
- 每章最后 1 题应为综合题（free_coding），作为章节 Boss

### 3.2 节

- 每节 5-7 题
- 节间线性解锁
- 难度分布：前 2 题 1-2★，中间 2-3★，**最后 1 题 ≥3★**
- 难度星级定义：

| 星级 | 判定标准 | 示例 |
|------|---------|------|
| 1★ | 单一知识点，直接套用语法 | `list.append(x)`、`d[key]` |
| 2★ | 单一知识点，需理解语义 | 负数索引、切片理解、in 判断 |
| 3★ | 两个知识点组合，或需条件判断 | 带 if 的推导式、嵌套索引、集合运算 |
| 4★ | 3+ 知识点综合，或需设计小算法 | free_coding 综合题、多条件判断 |

### 3.3 备用题

- 每个 knowledgeTag 配 2 道备用题
- 备用题不能再用 input()
- 备用题的 `knowledgeTags` 至少 1 个 tag 与原始题重叠

## 四、容器测试限制（pyodide）

以下 Python 特性在 Pyodide 中不可用：

| 特性 | 原因 | 处理方式 |
|------|------|---------|
| `input()` | 浏览器无 stdin | **自动判题题中禁止使用**。仅人工验证题可用，且 validationMode 设为 `contains` |
| `ruff` / `uv` | 需要 Rust 运行时 | 仅出概念题（choice） |
| 网络请求 `requests` | Pyodide 沙箱限制 | 仅出概念题 |
| 文件读写 `open()` | 无文件系统 | 用内存操作替代 |

**input() 规则总结：**

```
自动判题（code_fill / code_fix / free_coding） → 绝对禁止 input()
人工验证题                                         → 可以用 input()，validationMode 用 contains
                                                   每章不超过 1 题
                                                   不计入 e2e 自动测试通过率
```

## 五、数据创建清单

生成新题目数据时逐项检查：

- [ ] TS 字符串无意外转义（检查所有含反斜杠的选项/代码）
- [ ] 选项互斥，没有多个"都对"的选项
- [ ] correctOption 在 options 范围内
- [ ] correctCode 能实际运行并产生 expectedOutput
- [ ] validationMode 适用于题目的输入模式
- [ ] free_coding 有 ≥4 个 testCases 覆盖边界
- [ ] initialCode 正确引导用户但不泄露答案
- [ ] explanation 解释正确/错误原因
- [ ] commonErrors 模式与实际常见错误匹配
- [ ] 3层 hint 完整（hint / hintRoleplay / hintDirect）
- [ ] knowledgeTags 与备用题库有重叠
- [ ] 数据文件格式符合 `types/index.ts` 的 `Question` 接口

## 六、智慧之书页面模板

每节对应一个智慧之书页面。每页必须包含以下结构：

```
标题                      — 知识点名称（如"列表：数据的容器"）
专业定义                  — 一句话定义
语法规则（≥2 条）         — 核心语法规则
法阵演示（可运行代码）    — 一个最小可运行示例，默认显示输出
设计哲学（可选）           — 为什么这样设计
常见错误（≥1 条）         — 常见错误代码 + 说明 + 诺克斯提示
即时验证（可编辑沙盒）    — 用户可修改代码并运行
```

## 附录 A：Question 类型字段清单

创建题目数据时必须填写的字段（来自 `types/index.ts`）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | ✅ | 唯一标识，格式 `s{章}_{序号}` |
| `type` | 'choice' \| 'code_fill' \| 'code_fix' \| 'output_predict' \| 'free_coding' | ✅ |
| `chapterId` | string | ✅ | 所属章节 ID |
| `title` | string | ✅ | 技术题目（短） |
| `narrativeTitle` | string | ✅ | 叙事题目 |
| `description` | string | ✅ | 题目描述 |
| `narrativeDesc` | string | ✅ | 叙事场景描述 |
| `difficulty` | 1 \| 2 \| 3 \| 4 | ✅ | 难度星级 |
| `knowledgeTags` | string[] | ✅ | 知识点标签 |
| `explanation` | string | ✅ | 答案解析 |
| `options` | string[] | choice/predict 时必填 | 选项列表 |
| `correctOption` | number | choice/predict 时必填 | 正确选项索引 |
| `initialCode` | string | code 类型时必填 | 初始代码（可编辑） |
| `correctCode` | string | code_fill/fix/free_coding 时必填 | 参考答案 |
| `expectedOutput` | string | code_fill/fix 时必填 | 正确输出的预期值 |
| `validationMode` | 'exact' \| 'contains' | code_fill/fix 时必填 | 判题模式 |
| `testCases` | TestCase[] | free_coding 时必填 | 测试用例（≥4 个） |
| `hint` | string | ✅ | 第一层提示 |
| `hintRoleplay` | string | ✅ | 诺克斯提示（口语化） |
| `hintDirect` | string | ✅ | 直接提示（接近答案） |
| `commonErrors` | CommonErrorConfig[] | 建议 | 常见错误配置 |
