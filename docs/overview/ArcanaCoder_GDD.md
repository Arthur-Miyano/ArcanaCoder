## 一、项目概述

**游戏名称**：ArcanaCoder: 智慧圣殿编年史  
**类型**：像素风编程学习 RPG / 教育游戏  
**平台**：Web 浏览器（纯前端）  
**核心一句话**：玩家扮演一位在魔法大陆寻求终极智慧的女性贤者，通过"施展魔法"（编写 Python 代码）来解锁区域、提升等级，而她的伙伴诺克斯（一只会说话的雪白猫头鹰）会在旁陪伴、提示与总结。

**设计原则**：
1. **先沉浸，后学习**：玩家首先感受到的是"我在一个魔法世界里成长"，其次才是"我在学 Python"。
2. **诺克斯是朋友，不是老师**：提示不是标准答案的宣读，而是一个"也踩过坑的朋友"在分享经验。
3. **错误是剧情的一部分**：答错不是失败，而是"咒语波动不稳"，是成长叙事的一环。

## 二、世界观

### 2.1 背景故事

在名为 **艾瑟瑞亚 (Aetheria)** 的魔法大陆上，魔法并非天赋，而是可以被解析、重组与优化的"古老代码"。千年前的贤者们将世界的底层法则编织成一种名为 **"源语"** 的神秘文字——它有着严格的语法、精确的类型与强大的逻辑。

你是一位年轻的女性贤者，听闻在大陆尽头矗立着 **智慧圣殿 (Sanctum of Wisdom)**，那里藏有终极智慧的秘密。但通往圣殿的道路被五大区域封锁，每个区域都由一种"源语法则"守护。只有掌握这些法则，才能一步步接近真理。

陪伴你的是 **诺克斯 (Nox)**，一只额头带有蓝色符文印记的雪白猫头鹰。它是你导师的遗物，也是这片大陆上最古老的几只"源码之灵"之一。它不会直接给你答案，但会在你困惑时歪着头，用朋友的语气提醒你："嘿，我当年也在这个地方栽过跟头。"

### 2.2 世界结构：五大区域

通往智慧圣殿的路径由五个主题区域组成，对应 Python 学习的五大模块：

| 区域 | 主题 | 视觉基调 | 氛围关键词 |
|------|------|---------|-----------|
| **语法圣殿** | 变量、类型、运算符、基础语法 | 古老石质殿堂，金色符文，幽暗蓝光 | 神秘、庄严、起源 |
| **爬虫山脉** | 字符串处理、数据清洗、文本操作 | 雪山连绵，数据流如极光般在天空舞动 | 冷冽、流动、探索 |
| **后端城** | 函数、模块、逻辑结构 | 蒸汽朋克高塔，齿轮与机械结构 | 精密、工业、秩序 |
| **数据库森林** | 列表、字典、数据结构 | 发光的树木，数据粒子如萤火虫般飞舞 | 生机、繁茂、连接 |
| **前端海岸** | 综合应用、项目实战 | 海边悬崖，代码如海浪拍打礁石 | 开放、创造、终章 |

**区域解锁规则**：线性推进。完成前一区域的全部"节"后，下一区域在地图上亮起。

## 三、角色设定

### 3.1 主角：贤者 (The Mage)

**身份**：玩家化身，一位寻求终极智慧的女性贤者。  
**年龄**：约 20 岁，年轻但坚定。  
**外貌**：
- 棕色长发，通常束起或自然垂落
- 绿色眼睛
- 身着深紫色（紫罗兰色 `#4B0082`）长袍
- 手持一根木制法杖，杖端镶嵌绿色宝石
- 整体气质：求知、沉静、略带书卷气，但眼神中有执着

**性格**：
- 不是天生的天才，而是靠反复试炼积累智慧的"苦修型"角色
- 答对时会微微跳跃，法杖发光；答错时会困惑地摇头，但不会气馁
- 升级时会被金色光芒包围，长袍无风自动

**动画状态**：
- **待机**：呼吸起伏，3帧循环，偶尔眨眼
- **答对**：开心表情，眼睛弯成弧线，法杖散发金色粒子，轻微跳跃
- **答错**：困惑皱眉，歪头，轻微摇头，2帧循环
- **升级**：长袍展开，金色光晕环绕，5帧循环
- **思考**：手托下巴，眨眼，2帧循环（用于加载/等待状态）

### 3.2 伙伴：诺克斯 (Nox)

**身份**：源码之灵，雪白猫头鹰，导师的遗物。  
**外观**：
- 通体雪白的羽毛（`#FFFAFA`）
- 金色眼睛，锐利但温和
- 额头中央有淡蓝色符文印记，在说话时会微微发光
- 体型较小，通常以 16×16 或 24×24 像素呈现

**性格**：
- **朋友，不是导师**。它不会说"你应该这样做"，而是说"我当年也踩过这个坑"
- 语气轻松，会使用口语化表达："嘿""哈哈""不对不对""等等"
- 会分享自己的"黑历史"来安慰玩家，比如："dict.get() 和 dict[] 不是一回事，我当年也搞混过，还被导师敲了脑袋。"
- 在玩家连续答错时，语气会变得疲惫但温柔："别急别急，我们慢慢来，这个知识点确实有点绕。"

**动画状态**：
- **待机**：栖息状态，转头张望，3帧循环
- **说话**：头部上下微动，额头符文发光，2帧循环
- **飞行**：翅膀扇动，4帧循环（用于转场/过渡）
- **警示**：翅膀张开，眼睛发光，警觉姿态（用于严重错误提示）

## 四、核心体验循环

### 4.1 玩家旅程

```
登录 → 选择今日学习时长 → 查看诺克斯建议 → 选择区域/节
  → 阅读"智慧之书"（概念讲解） → 进入试炼（答题）
    → 提交魔法（代码/选择） → 获得即时反馈
      → 正确：经验+ 魔力共鸣 → 下一试炼
      → 错误：咒语波动不稳 → 诺克斯提示 → 重试/补练
        → 完成一节：升级判定/通关动画 → 返回地图
```

### 4.2 关键体验节点

| 节点 | 体验目标 | 设计要点 |
|------|---------|---------|
| **选择时长** | 建立掌控感 | 诺克斯根据时长生成今日计划，让玩家感觉"有人在帮我安排" |
| **智慧之书** | 降低学习焦虑 | 3D 翻页效果，专业定义+语法规则+设计哲学+即时验证，像一本真正的魔法书 |
| **试炼中** | 保持心流 | 无打断，即时反馈，5种题型轮换避免疲劳 |
| **答错时** | 消除挫败感 | 不是红叉，而是"咒语波动不稳"；诺克斯出现，用朋友语气解释；显示 diff 而非单纯判错 |
| **答对时** | 给予正反馈 | 贤者跳跃动画，金色粒子，经验条增长，诺克斯简短夸奖 |
| **完成一节** | 里程碑感 | 准确率统计，解锁下一节，地图节点从灰色石头变为金色星星 |
| **连续答错3次** | 防崩溃机制 | 诺克斯切换为"疲劳版本"，语气更温柔，提示更直接，可选降低难度 |
| **单节≥7题** | 防疲劳机制 | 弹出休息提示（羊皮卷风格），默认停留5秒，可跳过 |

## 五、叙事规范

### 5.1 术语替换表（强制）

以下术语在**所有 UI 文案、题目描述、反馈提示、按钮文字**中必须替换，禁止出现直白的技术教育用语：

| 禁用词 | 替换词 | 使用场景 |
|--------|--------|---------|
| 题目 / 习题 | **试炼** | "第3道试炼""完成本节试炼" |
| 答题 / 作答 | **施展魔法** / **编织咒语** | "施展你的魔法""提交咒语" |
| 正确答案 | **魔力共鸣** / **法则认可** | "魔力共鸣！你的源语很精准" |
| 错误 / 答错 | **咒语波动不稳** / **法则排斥** | "咒语波动不稳，再试一次？" |
| 代码 | **源语** / **咒文** | "写下你的源语""这段咒文有瑕疵" |
| 运行 / 执行 | **吟唱** / **释放** | "吟唱咒文查看效果" |
| 输出 / 打印 | **显现** / **回响** | "源语显现的结果" |
| 变量 | **符文容器** / **魔力印记** | （可选，视情境） |
| 字符串 | **真言** | （可选，视情境） |
| 学习 / 练习 | **修行** / **试炼** | "今日修行" |
| 知识点 | **法则** | "这条法则你已掌握 80%" |
| 技能树 | **奥术图谱** | "查看你的奥术图谱" |
| 等级 | **位阶** | "贤者位阶 Lv.3" |
| 经验值 | **魔力值** | "魔力值 +20" |

### 5.2 题目包装规范（强制）

每道试炼必须包含三个叙事层字段：

- **`narrativeTitle`**：魔法情境化的标题，而非技术标题
  - ❌ "字符串拼接"
  - ✅ "真言编织：将两个符文合二为一"

- **`narrativeDesc`**：将技术问题包装成魔法任务
  - ❌ "请使用 f-string 格式化输出"
  - ✅ "贤者，你需要将勇者的名字和战绩编织成一段预言。使用源语中的'真言编织术'（f-string）来显现这段文字。"

- **`narrativeExplanation`**：解析也必须是叙事化的
  - ❌ "f-string 是 Python 3.6 引入的..."
  - ✅ "真言编织术允许你在真言中直接嵌入符文容器的值，只需在大括号中写下容器名。这是圣殿中最常用的显现技巧之一。"

### 5.3 诺克斯提示三级规范

所有试炼的提示必须按以下三级准备，根据玩家状态递进：

| 级别 | 触发条件 | 语气要求 | 示例 |
|------|---------|---------|------|
| **hint** | 首次请求提示 | 简短、神秘、引导思考 | "试试看看这段源语中，哪个容器的名字拼写不太对？" |
| **hintRoleplay** | 第二次请求/答错1次 | 朋友语气，带个人经历 | "嘿，我当年也在这里卡过。你注意看那个引号，是不是一边用了单引号，一边用了双引法？哈哈，很常见的。" |
| **hintDirect** | 连续答错2次+ | 直接给出关键线索，但仍用世界观词汇 | "直接告诉你吧：这里的容器类型是'数'（int），不是'真言'（str）。试试把引号去掉？" |

**疲劳版诺克斯**（连续答错≥3次）：
- 语气更慢、更温柔，带鼓励
- 示例："别急别急，这个法则确实有点绕。我们先深呼吸一下...你看，这里的 dict.get() 其实是个安全的取法，就算容器里没有这个钥匙，它也不会报错，而是给你个'无'（None）。是不是比你直接用 dict[] 温柔多了？"

## 六、视觉与美术风格

### 6.1 整体风格

- **像素画 (Pixel Art)**，16-bit 复古风格
- **深色模式为主**，背景以深蓝/深紫为基调 (`#1a1a2e`)
- **魔法光效**：金色 (`#c9a227`) 和蓝色 (`#4a90e2`) 作为强调色
- **UI 边框**：白色细边框 + 轻微圆角（后期替换为羊皮卷纹理）

### 6.2 占位符规范（Phase 1-3）

在像素画资源生成前，使用以下纯色占位，但**必须让AI理解这些占位符代表什么**：

| 元素 | 占位样式 | 后期替换目标 |
|------|---------|------------|
| 贤者头像 | 32×32px 紫色圆 `#4B0082` + 金色边框 `#c9a227` | 贤者像素画精灵 |
| 诺克斯头像 | 16×16px 白色圆 `#FFFAFA` + 淡蓝边框 | 诺克斯像素画精灵 |
| 地图节点-已通关 | 16×16px 金色方块 | 金色星星像素画 |
| 地图节点-未通关 | 16×16px 灰色方块 `#666` | 灰色石头像素画 |
| 地图节点-当前 | 16×16px 蓝色方块，带呼吸动画 | 蓝色水晶像素画 |
| 技能树节点 | 12×12px 圆点 | 发光节点图标 |
| 背景 | 纯色 `#1a1a2e` | 场景背景图（按区域） |

### 6.3 场景背景（Phase 4）

每个区域答题时，背景应切换为对应场景：
- **语法圣殿**：古老石殿，金色符文悬浮，幽暗神秘
- **爬虫山脉**：雪山连绵，数据流如极光舞动
- **后端城**：蒸汽朋克高塔，齿轮转动，机械结构
- **数据库森林**：发光树木，数据粒子如萤火虫
- **前端海岸**：海边悬崖，代码如海浪拍打礁石

## 七、音效与反馈风格（概念）

> 注：Phase 1-3 可无音效，但需预留设计空间。

| 事件 | 风格描述 |
|------|---------|
| 魔力共鸣（答对） | 清脆的水晶共鸣声，伴随轻微风铃尾音 |
| 咒语波动不稳（答错） | 低沉的嗡鸣，像魔法护盾被轻微撞击，不刺耳 |
| 升级 | 庄严的钟声 + 金色粒子散落声 |
| 诺克斯说话 | 轻微的羽毛摩擦声 + 猫头鹰低鸣作为提示音 |
| 翻页（智慧之书） | 羊皮纸摩擦声 |
| 地图节点解锁 | 水晶点亮的声音，由近及远 |

## 八、与《设计规格.md》的衔接

本文件（GDD）回答 **"做什么、什么体验、什么感觉"**。《设计规格.md》回答 **"怎么实现、什么结构、什么接口"**。两者关系如下：

| GDD 定义 | TDD 实现 |
|---------|---------|
| 贤者答对时跳跃 | `mage-sprite` CSS 动画，切换答对帧 |
| 诺克斯朋友语气 | `hintRoleplay` 字段内容规范 + LLM 润色 prompt |
| 术语替换表 | 所有 UI 文案走 `i18n` 或常量表，禁止硬编码禁用词 |
| 题目三层叙事 | `Question` 接口必须含 `narrativeTitle/narrativeDesc/narrativeExplanation` |
| 疲劳控制 | `QuizPanel` 的 `consecutiveWrong` 计数器 + 诺克斯疲劳文本切换 |
| 五大区域 | `Chapter` 数据结构 + 背景图切换逻辑 |

**冲突裁决**：当技术实现（如性能优化、组件复用）与游戏体验（如叙事沉浸、角色一致性）冲突时，**优先保障体验**。技术方案应围绕体验目标调整，而非反之。

## 九、给 AI 编码助手的执行摘要

每次生成代码/文案/题目时，请自检：

1. [ ] 这段文案是否使用了"试炼/源语/魔力共鸣/咒语波动不稳"等世界观词汇？
2. [ ] 诺克斯的提示是否像"会犯错的朋友"，而不是"全知的客服机器人"？
3. [ ] 题目是否有 `narrativeTitle` 和 `narrativeDesc` 包装，而非直白的技术描述？
4. [ ] 错误反馈是否给了玩家"再试一次"的勇气，而不是挫败感？
5. [ ] UI 文本是否避免了现代编程教育用语（如"运行代码""查看输出"）？
