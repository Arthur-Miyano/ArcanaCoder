## 一、项目概况

纯前端像素风 Python 学习游戏。Vue 3 + Vite + TypeScript + Pyodide（浏览器内运行 Python）。
部署：GitHub Pages。存储：IndexedDB（idb 库）。

## 三、内容结构

### 3.1 阶段（Chapter）

| 区域 | 章 | 名称 | 题量 | 前置 | 状态 |
|------|----|------|------|------|------|
| 语法圣殿 | 1 | 变量与类型 | 28 题（4 节） | 无 | ✅ 数据就绪 |
| 爬虫山脉 | 2 | 字符串与输入输出 | 20 题（3 节） | 第 1 章 | ✅ 数据就绪 |
| 爬虫山脉 | 3 | 列表、元组、字典与集合 | 24 题（4 节） | 第 2 章 | ✅ 数据就绪 |
| 爬虫山脉 | 4 | 条件判断与循环 | 22 题 | 第 3 章 | 🔲 待生成 |
| 后端城 | 5 | 函数与模块 | 20 题 | 第 4 章 | 🔲 待生成 |
| 后端城 | 6 | 异常处理与 pathlib | 20 题 | 第 5 章 | 🔲 待生成 |
| 数据库森林 | 7 | 面向对象 | 22 题 | 第 6 章 | 🔲 待生成 |
| 数据库森林 | 8 | 类型注解、dataclass 与上下文管理器 | 20 题 | 第 7 章 | 🔲 待生成 |
| 前端海岸 | 9 | pytest、logging 与 ruff 规范 | 18 题 | 第 8 章 | 🔲 待生成 |
| 前端海岸 | 10 | async/await、uv 与打包发布 | 18 题 | 第 9 章 | 🔲 待生成 |

- 第 9-10 章为高阶内容：以概念题为主（choice 70% + code_fill 30%），ruff/uv/打包 因 Pyodide 无法运行，仅出概念题+智慧之书
- 后续扩展：SQL 基础、Web 框架（FastAPI）可作为新区域追加

### 3.2 节（Section）结构

每阶段拆为 3-4 节，每节 5-7 题。节间线性解锁。

```
第 1 章：变量与类型
├── 第 1 节：变量赋值（7 题）— 无前置
├── 第 2 节：类型系统（7 题）— 前置：第 1 节完成
├── 第 3 节：运算符与格式化（7 题）— 前置：第 2 节完成
└── 第 4 节：综合应用（7 题）— 前置：第 3 节完成
```

### 3.3 备用题库

每知识点（knowledgeTag）配 2 道备用题。每节最多插入 3 道备用题。
答错时从备用题库抽同知识点题目插入当前节末尾。

### 3.4 疲劳控制

| 条件 | 行为 |
|------|------|
| 单节答题数 ≥ 7 | 弹出休息提示（可跳过，默认停留 5s） |
| 连续答错 ≥ 3 | 诺克斯提示变为疲劳版本 |

```typescript
interface Question {
  id: string
  type: QuestionType

  stage?: string
  technicalDesc?: string

  commonErrors?: CommonErrorConfig[]

```

### 4.2 Chapter + Section

```typescript
interface ChapterSection {
  id: string
  name: string          // 技术名（如"字符串进阶"）
  displayName: string   // 叙事名（如"字符裂解"）
  questionIds: string[]
  unlockAfter?: string
}

interface Chapter {
  id: string
  name: string          // 技术名（如"第 2 关：字符串与输入输出"）
  displayName: string   // 叙事名（如"爬虫山脉"）
  description: string
  sections: ChapterSection[]
  regionName: string    // 所属区域（如"爬虫山脉"）
  regionOrder: number   // 区域在地图上的顺序
}
```

### 4.3 SectionProgress（Store）

```typescript
interface SectionProgress {
  completed: boolean
  questionResults: Record<string, { correct: boolean; score: number; attempts: number }>
  consecutiveWrong: number
}
```

## 十一、AI vibe coding 预防规范

> 本项目大量使用 AI 辅助生成代码。为避免 AI 生成代码常见的虚假引用、过度耦合、重复定义、虚假测试等问题，所有贡献者（包括 AI）必须遵守以下规范。

### 11.1 数据完整性（最高优先级）

- **题目 ID 必须全局唯一**。新增/复制题目后，必须运行 `npm run test` 中的数据完整性检查。
- **章节/节/题目关联必须一致**。`chapterId`、`sectionId`、`questionIds` 三者必须互相匹配。
- **新增章节时**，必须同步更新 `questions.ts`、`wisdom.ts`、章节显示名映射（`ChapterSelect.vue`）。
- **推荐**：在 CI 中加入题目数据校验脚本，检查 ID 唯一性和关联完整性。

### 11.2 组件架构

- **单一职责原则**：Vue 单文件组件超过 250 行必须拆分。
- **禁止上帝组件**：组件不应同时承担状态管理、业务逻辑、数据校验、UI 渲染等多种职责。
- **推荐拆分方式**：
  - UI 布局 → 组件本身
  - 状态机/流程控制 → composable（如 `useQuizEngine`）
  - 数据校验/执行 → service（如 `services/questionValidation.ts`）
  - 游戏规则 → `constants/` + `utils/`

### 11.3 Store 封装规范

- **禁止组件直接访问 `store.state`**。
- Store 必须通过 `computed` 暴露 getter，或通过函数暴露 action。
- 需要新增状态访问时，先在 store 中定义 getter，再在组件中使用。
- Store action 若涉及异步保存，必须声明为 `async` 并 `await`。
- `store.state` 目前仍保留在 return 中（组件未直接使用，E2E 测试的 unlockStore 需要直接修改多字段状态）。
  测试中如需访问内部状态，首选 `store.$state`（Pinia 内置只读访问），或通过公开的 getter/action。

```ts
// ✅ 推荐
const unlockedChapters = computed(() => state.value.unlockedChapters)
function isChapterUnlocked(id: string): boolean { return state.value.unlockedChapters.includes(id) }

// ❌ 禁止在组件中写
store.state.unlockedChapters.includes(id)
```

### 11.4 异步处理规范

- 所有调用 `save()`、`load()`、Pyodide 运行、网络请求的地方必须使用 `await` 或 `.catch()` 处理。
- 禁止 fire-and-forget 式异步调用。
- `async` 函数必须显式标注返回类型 `Promise<void>` 或 `Promise<T>`。

```ts
// ✅ 推荐
async function submitAnswer(...): Promise<{ leveledUp: boolean; expGained: number }> {
  // ...
  await save()
  return { leveledUp, expGained }
}

// ❌ 禁止
function submitAnswer(...) {
  save() // 未 await，错误无法捕获
}
```

### 11.5 类型安全

- **禁止 `any`**。确实无法确定类型时使用 `unknown` + 类型守卫。
- **禁止 `!` 非空断言**。应通过类型收窄或运行时校验处理。
- 函数返回类型必须显式声明，尤其是导出函数和 store action。
- 数据模型应使用 discriminated union 表达互斥字段，避免大量可选属性。
- 当前已知的 `any` 缺口（`userAnswer: ref<any>`、`window as any`）已在 `已知缺口.md` 中追踪。
  核心判断：`userAnswer` 因三个组件的 v-model 类型不兼容无法用联合类型替代，属于类型系统能力边界。

```ts
// ✅ 推荐
type Question = ChoiceQuestion | CodeQuestion | ...

// ❌ 避免
interface Question {
  options?: string[]
  correctOption?: number
  initialCode?: string
  // 所有字段都可选，无法保证类型安全
}
```

### 11.6 游戏规则与算法集中管理

- 所有 magic numbers / magic strings 必须抽到 `constants/` 或 `utils/`。
- 包括但不限于：
  - 经验值表（`EXP_TABLE`）
  - 升级公式（`calcExpToNext`）
  - 答错惩罚比例
  - 掌握度算法（`calcMastery`）
  - 疲劳阈值（如 7 题、3 错、45 分钟）

### 11.7 可复用逻辑抽离

- 多个组件中重复出现的模式应抽成 composable 或 util：
  - Python 运行 → `usePythonRunner`
  - 代码编辑器配置 → `useCodeMirror`
  - 通用 UI 组件 → `BaseButton`、`AvatarIcon` 等

### 11.8 测试规范

- **禁止虚假测试**：断言必须能真正失败。
- 避免以下弱断言：
  - `expect(x).toBeTruthy()`
  - `expect(arr.length).toBeGreaterThanOrEqual(0)`
- ⚠️ `expect(x).toBeDefined()` 在 **数据完整性测试中可接受**（验证 ID/引用存在性），
  但在行为测试中仍应避免。使用时必须附带自定义错误消息方便定位。
- 每个被测函数的核心路径、边界路径都必须覆盖。
- 新增 store 方法必须新增对应测试。
- 新增数据字段必须新增数据完整性测试。
- E2E 测试优先使用 `data-testid`，避免硬编码中文文案。

### 11.9 代码审查流程

每次 AI 生成代码后，必须执行：

1. `npm run typecheck`
2. `npm run test`
3. `npm run build`
4. `git diff` 人工审查
5. 检查是否有新增 `any`、未 await 的异步调用、硬编码值
6. 阅读所有新增/修改的测试断言，确认其有效性

### 11.10 提交前检查清单

- [ ] 没有死导入 / 死代码
- [ ] 没有 `.bak` 等临时文件进入仓库
- [ ] 没有全局函数调用（如 `confirm()`），使用 `window.confirm` 并检查环境
- [ ] 没有除零等边界错误（如 `x / total` 改为 `total ? x / total : 0`）
- [ ] 没有重复定义的 ID / 函数 / 常量
- [ ] Store 没有直接暴露 `state` 给组件
- [ ] 异步函数正确 `await`
