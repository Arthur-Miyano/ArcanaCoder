## 二、修复方案

### P0-1：选项显示修复

**涉及文件：** `ChoiceQuestion.vue`、`OutputPredict.vue`

**方案：** `{{ opt }}` → `<span class="whitespace-pre-wrap">{{ opt }}</span>`

**影响范围确认：** 两组件均直接渲染 `{{ opt }}`，无子组件/插槽/排序逻辑。数据中 `\n` 为真实换行符——HTML 默认将其折叠为空格，`whitespace-pre-wrap` 可保留该换行。

**数据层补丁：** 后续题目生成时统一清洗规则：选项如需展示转义符效果用 `\\n`（双反斜杠），如需展示换行效果用真实 `\n` + UI 层 `white-space` 控制。

### P0-2：答案数据正确性核查

#### Step A — 静态核查（Node.js 脚本）

对每条题目：

```
choice / output_predict:
  assert correctOption >= 0 && correctOption < options.length
  assert options[correctOption] !== undefined && options[correctOption] !== ''
  assert 组件渲染顺序 === 数据原始顺序 (ChoiceQuestion.vue 和 OutputPredict.vue 均用 v-for 直接遍历 question.options，无 sort/shuffle/computed 重排 — 已验证代码，顺序一致)

code_fill / code_fix:
  assert expectedOutput !== undefined
  assert correctCode !== undefined

free_coding:
  assert testCases 数组非空
  for each testCase:
    assert testCase.input 字段存在
    assert testCase.expected 字段存在且非空
    assert testCase.input/expected 类型匹配
```

#### Step B — 动态验证（Pyodide + Playwright PoC）

加载 Pyodide，对每条 `code_fill` / `code_fix` 执行 `correctCode`，比对输出与 `expectedOutput`，输出勘误表。

#### Step C — 长期改进建议（P2 技术债）

将 `correctOption` 从索引改为选项文本哈希，彻底规避顺序问题。当前不改，工作量大。

### P0-3：开场介绍

**涉及文件：** 新增 `IntroOverlay.vue`

**方案：**
1. LoadingOverlay ready 后、DailyPlanDialog 之前，检查 `introShown` 标记
2. 未看过 → 全屏介绍（世界观简述），底部"跳过"按钮

4. 无痕模式下 IndexedDB 读取失败 → 默认显示介绍（新用户大概率）

### P1-4：疲劳计数器修复

#### Step A — 验证（浏览器控制台）

打开 Chrome DevTools → Console，粘贴以下任意一种：

**方案一（项目使用 idb 库，可用 Promise 语法）：**
```javascript
// 项目依赖 idb 库（storage.ts），但控制台中没有 import，用原生 API 确保可执行
const request = indexedDB.open('ArcanaCoder', 1);
request.onsuccess = () => {
  const db = request.result;
  const tx = db.transaction('gameState', 'readonly');
  const store = tx.objectStore('gameState');
  const getReq = store.get('main');
  getReq.onsuccess = () => {
    const record = getReq.result;
    const sp = record?.state?.sectionProgress || {};
    console.log('sectionProgress keys:', Object.keys(sp));
    // 检查每个 section 的 questionResults 数量——是否远超用户记忆中的做题数
    for (const [secId, sec] of Object.entries(sp)) {
      const qCount = Object.keys((sec).questionResults || {}).length;
      if (qCount > 0) console.log(`${secId}: ${qCount} 条答题记录`);
    }
  };
};
```

**排查方向说明：** `fatigueQuestionCount` 是 `ref(0)`（QuizPanel.vue 第 38 行），纯组件级、不持久化，IndexedDB 不可能有它的残留值。问题不在存档，而在**计数器累加逻辑**：
- `submit()` 中 `fatigueQuestionCount.value++` 是否被重复调用（比如备用题插入 + 原题提交计为两次）
- 同一道题在重试模式下是否重复计数
- 或是 `sectionProgress` 里累积了远超实际做题数的 `questionResults` 条目（旧版数据的 `completedQuestions` 数组残留）

#### Step B — 修复

**前置确认：** `watch(currentSection, ...)` 目前的完整职责：
1. `questions.value = getQuestionsBySection(sec.id)` — 加载当前节题目
2. `currentIndex.value = 0` — 重置题目索引
3. `backupLimit.value = 0` — 重置备用题计数
4. `usedBackupIds.value = new Set()` — 重置备用题已用集合
5. `fatigueQuestionCount.value = 0` — 重置疲劳计数
6. `fatigueConsecutiveWrong.value = 0` — 重置连续错误计数

**决策：** watch 职责内聚，拆出单一逻辑反而增加耦合度。改为在 `selectSection()` 调用处显式触发重置，避免 watch 在组件复用场景下不触发的问题。

### P1-5：自动化测试

**技术栈：** Playwright

#### 风险 1：Pyodide CDN 在 Headless 模式下的加载

**现状：** `pyodide.ts:3` 硬编码从 CDN 加载：`https://cdn.jsdelivr.net/pyodide/v0.27.4/full/`
**风险：** Playwright headless + CI 环境下 CDN 可能不可用/CORS 受限/加载超时

**验证结果：** 代码中无本地 fallback，vite.config.ts 无 proxy 配置

**方案：** 下载 Pyodide 到本地 `public/pyodide/`，测试时 `pyodide.ts` 通过环境变量切换 indexURL：
```typescript
// pyodide.ts
const PYODIDE_BASE = import.meta.env.DEV
  ? '/pyodide/'          // 开发环境用本地
  : 'https://cdn.jsdelivr.net/pyodide/v0.27.4/full/'  // 生产用 CDN
```
Playwright 中先用 `headless: false` 本地调通，再切 headless。

#### 风险 2：组件选择器策略

**现状：** 全局搜索 `data-testid` → **0 个命中**。现有组件无法被 Playwright 可靠定位。

**方案：** 在关键组件中加 `data-testid` 属性（与 P0-1 修复同步进行）：

| 组件 | 加 testid 的元素 | testid 值 |
|------|-----------------|-----------|
| ChoiceQuestion | 选项按钮 | `choice-option-{index}` |
| OutputPredict | 选项按钮 | `choice-option-{index}` |
| QuizPanel | 提交按钮 | `btn-submit` |
| QuizPanel | 下一题按钮 | `btn-next` |
| FeedbackPanel | 反馈面板容器 | `feedback-panel` |
| CodeQuestion | 代码编辑器 | `code-editor` |
| NoxDialog | 诺克斯对话气泡 | `nox-message` |

#### 风险 3：异步状态等待

**现状：** Pyodide 执行代码是异步的（`await runPython(code)`），但组件中无明确的"执行完成"DOM 标记。Playwright 无法可靠判断"执行完毕、结果已渲染"。

**方案：**
1. QuizPanel 提交按钮上加动态 `data-executing`：
```html
<button
  :data-executing="submitting ? 'true' : 'false'"
  data-testid="btn-submit"
  ...
>
```
2. Playwright 中可靠等待：
```typescript
await page.click('[data-testid="btn-submit"]');
await page.waitForSelector('[data-executing="false"]');
await expect(page.locator('[data-testid="feedback-panel"]')).toBeVisible();
```

#### PoC 先行（第 4 步）

最小验证链路：
1. Playwright 启动浏览器 → 加载页面
2. 等待 Pyodide ready（loading 页消失）
3. 关闭每日计划弹窗
4. 进入第 1 章 → 第 1 节 → 回答一个 choice 题
5. 断言反馈面板显示"魔力共鸣"
6. 完成后退出

#### 配置要点
- `playwright.config.ts` → `timeout: 30000`
- Playwright 的 `page.goto()` URL 必须匹配 vite 的 `base` 配置。当前 `vite.config.ts` 中 `base: '/ArcanaCoder/'`，所以 Playwright 应打开 `http://localhost:5173/ArcanaCoder/`，而非根路径
- 本地用 `headless: false` 调通后再切 headless

#### 分阶段
- Phase 1（核心）：PoC + data-testid + data-executing → 覆盖各题型答题链路
- Phase 2（边缘）：备用题触发 → 混合对错 → 重试模式 → 疲劳提示

- [vite-vue-ultimate-starter](https://github.com/kaandesu/vite-vue-ultimate-starter)（25★）— Playwright + Vitest + Vue3 集成结构

### 遗漏项修复

| # | 措施 | 实现方式 |
|---|------|---------|
| A | 错误监控 | `app.config.errorHandler` + `window.onerror` 双保险。`window.onerror` 只捕获同步运行时错误，Vue 组件渲染异常（模板错误、生命周期报错）需 `app.config.errorHandler` 才能捕获。两者同时注册 |
| B | 性能基线 | LoadingOverlay 记录 Pyodide 加载完成耗时，超 10s 在界面底部显示"源语编译器加载中…（超过 10 秒请检查网络）"，而非仅控制台警告（用户看不到控制台） |
| C | 数据版本 | gameStore `createInitialState` 中加 `version: 1`，load 时做版本兼容 |
| D | 移动端 | 后续验证时再覆盖，当前不做改动 |
