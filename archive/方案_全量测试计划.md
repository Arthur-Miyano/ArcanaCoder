## 阶段 2：场景 A（全对验证）— 按节拆分

### 执行方式

每节一个独立测试文件，一个 `test()` 内跑完该节所有题。

```
e2e/tests/
  scenario_A_s1_vars.spec.ts      # 5 题 ✅ 已完成
  scenario_A_s1_types.spec.ts     # 4 题
  scenario_A_s1_ops.spec.ts       # 10 题
  scenario_A_s1_strings.spec.ts   # 9 题
  scenario_A_ch2_strings.spec.ts  # 20 题
```

### 数据来源

每节对应一个 `e2e/fixtures/{section}.json`。数据从源码提取。

提取方式：三个数据文件（stage1/2/backup_questions.ts）都是纯数组导出，`import type` 在运行时无依赖。用 `tsx` 脚本执行提取。

### 产出

每节输出独立结果 JSON，最后合为 `scenario_A_report.json`。

```
npx playwright test e2e/tests/scenario_A_*.spec.ts --workers=4
```

预计执行时间：5-8 秒/题 × 84 题 ÷ 4 workers ≈ 1.5-3 分钟。

并发注意：4 个 workers = 4 个浏览器实例同时加载 Pyodide，每个 WASM 编译约占用 200-400MB 内存。首次运行需观察内存占用，超过 80% 降为 workers=2。

### 节失败时的行为

单节失败不影响其他节。结果报告中标记`{ section, passed, failed, details }`。

## 阶段 3：场景 B（全错验证）— 每节只跑 3 题

### 控制备用题膨胀

每节只跑前 3 题，全部答错。备用题触发机制：**每答错一题触发一道**（非连续答错），最多 3 道（`backupLimit.value < 3`）。

- 第 1 题答错 → 触发备用题 1
- 第 2 题答错 → 触发备用题 2
- 第 3 题答错 → 触发备用题 3（达到上限）
- 最终总题数 = 3(原题) + 3(备用上限) = 6

### 总题数读取方式

不依赖 Vue 组件内部状态。直接在页面元素中读取——QuizPanel 的题号显示为 `{{ currentIndex + 1 }}/{{ questions.length }} 题`（如"1/5 题"、"3/8 题"）。Playwright 读取该文本并解析出 `questions.length`。

### 范围

选 3 节执行（涵盖 choice 和 code 题）：

```
s1_vars (5 题，跑前 3) → 触发 3 道备用题
s1_ops (10 题，跑前 3) → 触发 3 道备用题
ch2_strings (20 题，跑前 3) → 触发 3 道备用题
```

## 阶段 4：场景 C（混合对错）— 验证重试

在 s1_vars 执行。备用题上限 3 道，每答错一题触发。

```
原始 5 题，答错模式为 [错, 错, 对, 错, 错]：
s1_01 答错 → backupLimit=1, 总题数=6
s1_02 答错 → backupLimit=2, 总题数=7
s1_03 答对 → 不触发, 总题数=7
s1_04 答错 → backupLimit=3, 总题数=8（达上限）
s1_05 答错 → backupLimit=3（已满）, 不触发, 总题数=8
最终断言：总题数 = 5 + 3(备用上限) = 8
```

总题数读取方式：同上，读 QuizPanel 的 `{{ currentIndex + 1 }}/{{ questions.length }}` 文本。

不扩展其他节，验证一次机制即可。

## 阶段 5：场景 D（代码变体）— 选 5 道代表性 code 题

validationMode=contains 确认存在：stage1 有 3 题（s1_23, s1_24, s1_25），backup 有 4 题。

| 题 ID | 判题模式 | 选择理由 |
|-------|---------|---------|
| s1_02 | exact | 基础 code_fill，字符串完全匹配 |
| s1_05 | exact | 字符串拼接，涉及多个运算符 |
| s1_23 | contains | validationMode=contains 的题 |
| s2_20 | testCase | free_coding，链式调用判题 |
| bk_fstring_1 | exact | f-string 格式化匹配 |

每道题执行 4 个变体：
```
1. 多余空格 → 预期正确或错误
2. 多余换行 → 预期正确
3. 变量名大小写 → 预期错误（diff 显示）
4. 等价写法（如 f-string 替代拼接）→ 取决于判题逻辑
```

执行后输出每道题每个变体的实际判题结果，用于判断 diff 引擎的容错边界。
