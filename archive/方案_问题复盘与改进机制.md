### 浪费的根因

1. **需求未确认就动手**：用户说"写自动化测试"，我默认选 Playwright，没问"这测试的目标是什么、输出什么、谁看"
2. **同一方向连败 2 次不回头**：R1-R4 都在修浏览器加载问题，没想过换方案（Node.js 直接读数据也能完成核心目标）
3. **诊断代码本身制造了问题**：加的 `isVisible()` + `textContent()` 干扰了 WASM 编译
4. **没有定义"完成"标准**：写测试写到第 20 版才明确"这个测试要干什么"

## 二、改进机制：以后怎么避免

### 2.1 动手前 checklist（每次任务开始时执行）

```
□ 需求：你要产出什么东西？格式是什么？谁用？
□ 选型：最小工具是什么？Node 脚本 < Vitest < Playwright
□ 完成标准：什么算"做完"？列出可检查的指标
□ 时间预算：如果 X 分钟内没跑通，停下来重评估方向
```

### 2.2 "2 败停"规则

同一个技术问题连续失败 2 次，必须停下来书面说明：
```
当前问题的根因是什么？
换一个方向能否绕过这个问题？
如果换方向，代价是什么？
```

由用户决定是继续深挖还是换方向。不再出现 R3-R6 连续 4 轮修同一个问题的情况。

### 2.3 虚假测试防范

测试必须满足以下至少一项才能算有效：
- 使用**真实数据**（直接从 `stage1_questions.ts` 读取，而非手动编写）
- 或使用**真实浏览器**执行（Playwright 打开页面）
- 或验证的是**独立逻辑**（不依赖数据正确性）

凡是用 mock 数据、硬编码输入、不接触真实数据源的测试，在报告中标记为「⚠️ 数据无关测试」。
