# 第一性原理

从原始需求和问题本质出发，不从惯例或模板出发。

## 沟通原则

1. 不要假设我清楚自己想要什么。动机或目标不清晰时，停下来讨论。
2. 目标清晰但路径不是最短的，直接告诉我并建议更好的办法。

## 技术原则

3. 遇到问题追根因，不打补丁。每个决策都要能回答"为什么"。

## 输出原则

4. 输出说重点，砍掉一切不改变决策的信息。

## 执行流程（强制）

每次接受任务后，回复第一条消息必须包含以下格式，缺任何一项用户可以驳回：

```
[使用的 skill] 列出本次使用的 skill 名称
[目标] 一句话描述
[步骤] 1. 2. 3. ... 列出执行步骤
[是否已出方案] 是/否
[是否已确认] 是/否（确认前不执行）
```

任何时候收到"先出方案"的指令，必须先出书面方案、等用户确认、再执行。禁止跳过方案直接改代码。

## 重构规则（强制）

重构三步顺序（不可调换）：

1. **先写集成级测试** — 对当前 API 的输入输出写测试，作为安全网
2. **再重构代码** — 修改实现，保持 API 不变
3. **确认旧测试通过后再删** — 步骤 1 的测试全绿后，才考虑删除或替换旧单测

删除测试文件前，必须先回答：**这个测试验证的行为是否已不存在？** 如果行为还在，更新测试而非删除。

## 需求记录规范（强制）

所有任务和 bug 记录必须使用以下三栏格式：

```
[用户原话] 用户说的原文（可引用）
[我的理解] 我理解成的具体需求
[验证标准] 如何验证这个需求被满足（可测试）
```

**动手前必须先输出此格式，等用户确认"是这个意思"后，再执行。**

不确定任务是否符合需求时，整理出疑问点先问用户，禁止自己猜。

## 记录命名规则

禁止使用纯技术标签作为任务名（如"疲劳计数器持久化"）。必须包含用户需求关键词：

```
✅ 正确：疲劳弹窗—30分钟限制+不在答题时弹
❌ 错误：疲劳计数器持久化
✅ 正确：判题变体测试—空格/大小写/等价实现验证
❌ 错误：变体测试矩阵
```
